# DeployGuard

**DeployGuard** is an AI-assisted **pre-deployment reliability analyzer** for Python
repositories. Point it at a Git repository and it will statically analyze the code and
its deployment configuration, reason about how the system is likely to behave in
production, simulate failure behavior under different load levels, and produce a
Site-Reliability-Engineer (SRE) style **reliability report** telling you whether the
system is safe to deploy — and whether a rollback should be recommended.

It combines classic **static analysis** (Python AST parsing, call-graph construction,
rule-based risk detection, infrastructure/config scanning) with a **LLM reasoning
pipeline** orchestrated by [LangGraph](https://github.com/langchain-ai/langgraph) and
served by [Groq](https://groq.com/).

## What it does

Given the URL of a target Git repository, DeployGuard:

1. **Clones** the repository locally (or `git pull`s and incrementally re-analyzes it if
   already cloned).
2. **Parses every Python file** into an AST and extracts functions, imports, calls, and
   "risky" operations (network / file / database access), noting whether risky calls
   happen inside loops or without `try/except` protection.
3. **Builds a repo-wide call graph** (nodes = functions, edges = resolved calls) with
   fan-in / fan-out metrics.
4. **Scans deployment/infra files** — `Dockerfile`, `docker-compose.yml`, Kubernetes
   manifests, and `requirements.txt` — for reliability anti-patterns (unpinned images,
   missing health checks, no resource limits, single replicas, unpinned dependencies,
   etc.).
5. **Runs a rules engine** that flags SRE risk categories (saturation, error-rate,
   single-point-of-failure, latency).
6. **Asks you targeted deployment questions** (peak RPS, replica count, caching, DB
   type, autoscaling, rollback strategy…) generated by the LLM based on the detected
   risks.
7. **Produces a reliability assessment** with a 0–100 reliability score, a risk level,
   and a list of failure points.
8. **Simulates** system behavior under best / average / worst-case load and decides
   whether **rollback is recommended**.
9. **Writes a final Markdown reliability report** (`reliability_report.md`).

## How it works (pipeline)

The pipeline is a LangGraph state machine (see `workflow.png`, generated on each run):

```
START
  │
  ▼
get_repo ──────────────┐
  │ (CLONED)           │ (already cloned → NOT CLONED)
  ▼                    ▼
generate_call_graph   pull_new_files   ← incremental re-analysis of changed files
  │                    │
  └────────┬───────────┘
           ▼
   inter_llm_response   ← LLM generates questions, YOU answer, LLM assesses reliability
           ▼
 architecture_extractor ← LLM builds a simulation-ready architecture model
           ▼
   simulation_engine     ← deterministic load/failure simulation (no LLM)
           ▼
   report_generator      ← LLM writes reliability_report.md
           ▼
          END
```

Two entry paths from `get_repo`:

- **First run (`CLONED`)** → `generate_call_graph` does a full analysis of the whole repo.
- **Subsequent runs (`NOT CLONED`, i.e. repo already on disk)** → `pull_new_files` runs
  `git pull`, detects changed files, and re-analyzes only those, splicing the results
  into the existing call graph and infra signals.

## Project structure

```
DeployGuard/
├── main.py              # Entry point: LangGraph pipeline orchestration + all graph nodes
├── ast_parser.py        # Python AST parsing → per-file JSON + repo-wide call graph builder
├── rules_engine.py      # SRE rule set run over the call graph + infra signals → risk flags
├── infra_parser.py      # Parsers for Dockerfile / docker-compose / K8s / requirements.txt
├── prompts/
│   ├── questions_prompt.txt      # Phase 1: generate targeted deployment questions
│   ├── assessment_prompt.txt     # Phase 2: full reliability assessment (score, risks)
│   ├── architecture_prompt.txt   # Build simulation-ready architecture model
│   └── closure_prompt.txt        # Final Markdown report format
├── requirements.txt     # Python dependencies
├── .env.example         # Template for required environment variables
├── .gitignore
└── workflow.png         # Auto-generated pipeline diagram (rewritten on each run)
```

Files/folders created at runtime (all git-ignored):

- `<repo-name>_clone/` — the cloned target repository.
- `build/` — all intermediate JSON artifacts (see [Output & artifacts](#output--artifacts)).
- `reliability_report.md` — the final report.

## Requirements

- **Python 3.9+** (the code uses modern type-hint syntax such as `set[str]`).
- A **Groq API key** (the LLM steps call Groq's `meta-llama/llama-4-scout-17b-16e-instruct`).
- **Git** installed and available on your `PATH` (used via GitPython to clone/pull).
- Network access to clone the target repo and reach the Groq API.

Python dependencies (from `requirements.txt`):

- `groq`, `langchain-groq` — LLM client
- `langgraph`, `langchain-core` — pipeline orchestration
- `GitPython` — clone/pull the target repo
- `PyYAML` — parse compose / Kubernetes manifests
- `json-repair` — repair malformed LLM JSON output
- `python-dotenv` — load `.env`
- `ipython` — used for the display/image helper

## Configuration

DeployGuard reads two environment variables from a `.env` file in the project root.
Copy the template and fill it in:

```bash
cp .env.example .env
```

Edit `.env`:

```dotenv
API_KEY = "your_groq_api_key_here"        # Groq API key (used for all LLM steps)
REP_URL = "https://github.com/owner/repo" # Git URL of the repository to analyze
```

| Variable  | Meaning                                                        |
|-----------|---------------------------------------------------------------|
| `API_KEY` | Your **Groq** API key. Loaded as `GROQ_API_KEY` in `main.py`. |
| `REP_URL` | The Git URL (HTTPS or SSH) of the target repo to analyze.     |

## Usage

Run the pipeline from the project root:

```bash
python main.py
```

What happens:

1. The target repo (`REP_URL`) is cloned into `./<repo-name>_clone/` (skipped if it
   already exists — then it `git pull`s and re-analyzes only changed files).
2. Static analysis runs and writes JSON artifacts into `./build/`.
3. The LLM prints **up to 5 deployment questions**. You answer them **interactively**:

   ```
   Questions:

   1. What is the expected peak requests per second?
   2. Is autoscaling enabled?
   3. How many application replicas are deployed?

   Provide answers in the same order.
   Write answers by separating them with a comma.

   > 500, no, 2
   ```

   Enter your answers on one line, **comma-separated, in order**.
4. The LLM produces a reliability assessment, the simulation runs, and the final report
   is generated.
5. `reliability_report.md` is written and printed to the console. `workflow.png` is
   (re)generated.

> Because step 3 requires typed input, run DeployGuard in an **interactive terminal**
> (not a fully non-interactive/CI context).

## Output & artifacts

### Final report
- **`reliability_report.md`** — the human-readable report. Sections: Reliability Summary
  (score, risk level, deployment status, rollback), Key Reliability Concerns, Critical
  Failure Points, Simulation Insights (best/average/worst case), Rollback Analysis,
  Recommendations, and a Final Verdict.

### Intermediate artifacts (`build/`)
| File | Produced by | Contents |
|------|-------------|----------|
| `<file>.py.json` | `ast_parser.py` | Per-file AST extraction (functions, imports, calls, risk tags) |
| `call_graph.json` | `build_call_graph()` | Repo-wide nodes/edges, unresolved calls, fan-in/out |
| `infra_signals.json` | `infra_parser.py` | Detected infra/config risks per file |
| `risk_factors.json` | `rules_engine.py` | Flat list of triggered SRE risk rules |
| `llm_response.json` | `inter_llm_response` | LLM reliability assessment (score, failure points) |

The simulation output (best/average/worst-case predicted reliability and rollback
decision) is held in pipeline state and summarized in the final report.

## How the analysis is done

### 1. AST parsing & risk tagging (`ast_parser.py`)
Each function is inspected for calls matching crude keyword lists:

- **network** — `requests`, `urlopen`, `socket`, `http`, `get`, `post`, …
- **file** — `open`, `read`, `write`, `remove`, `mkdir`, …
- **db** — `execute`, `cursor`, `commit`, `query`, `insert`, `select`, `find`, …

For every call it records whether it happens **inside a loop** and whether it is wrapped
in **try/except**. Functions get summary flags like `calls_made_inside_loop` and
`has_unprotected_risky_call`. `build_call_graph()` then merges all per-file JSONs into a
single graph and computes **fan-in / fan-out**.

### 2. Rules engine (`rules_engine.py`)
Independent rules flag:

- `unbounded_loop_with_risky_call` → **saturation** (risky call in a loop)
- `unprotected_risky_call` → **error_rate** (risky call with no try/except)
- `high_fan_in_risky_function` → **single point of failure** (fan-in ≥ `FAN_IN_THRESHOLD`, default 5)
- `unresolved_external_risky_call` → **latency/error_rate** (external library calls that touch network/db/file)
- `infra_config_risks` → aggregates all deployment-config risks, bucketed CRITICAL/HIGH/MEDIUM

### 3. Infra parsing (`infra_parser.py`)
- **Dockerfile** — unpinned base image, missing `HEALTHCHECK`, runs as root.
- **docker-compose** — missing restart policy, no resource limits, no healthcheck, weak `depends_on`.
- **Kubernetes** — single replica, no RollingUpdate, no resource limits, missing liveness/readiness probes, unpinned images.
- **requirements.txt** — unpinned dependencies (no `==`).

### 4. LLM reasoning (Groq + LangGraph)
Prompts in `prompts/` drive four LLM steps: **question generation**, **reliability
assessment**, **architecture modeling**, and **final report writing**. Robust JSON
extraction (`extract_json`, `pick_block`, `invoke_for_json`) tolerates code fences and
malformed output, retrying with the parse error fed back to the model.

### 5. Deterministic simulation (`simulation_engine`, no LLM)
Failure points are scored by **severity impact**, **probability weight**, and an
**activation load threshold** across three workload scenarios (best 1×, average 3×,
worst 8× baseline). Predicted reliability maps to a status
(Stable / Degraded / Partial Outage Risk / High Outage Risk), and **rollback is
recommended when worst-case predicted reliability < 60**.

## Troubleshooting

- **Auth / empty LLM output** → check `API_KEY` in `.env` is a valid Groq key and that
  the model name is available to your account.
- **Clone fails** → verify `REP_URL`, that Git is installed, and that you have access to
  the repo (SSH keys / HTTPS credentials).
- **"Repo already exists locally, skipping clone."** → DeployGuard switches to the
  incremental `pull_new_files` path. Delete `./<repo-name>_clone/` (and optionally
  `./build/`) to force a full fresh analysis.
- **Stuck at the questions prompt** → the run is waiting for your comma-separated
  answers on stdin; make sure you're in an interactive terminal.

## Notes & limitations

- Analysis targets **Python** code; non-Python source is only inspected as infra/config.
- Risk tagging is **keyword-based** and call resolution is **name-based** (exact name or
  last dotted segment), so cross-file/import resolution is approximate — external and
  ambiguous calls are deliberately kept as `unresolved_calls`.
- LLM outputs are **non-deterministic** in wording (though temperature is 0); scores can
  vary between runs, especially with different answers to the deployment questions.
- The reliability score and simulation are **advisory** decision-support signals, not a
  guarantee of production behavior.
